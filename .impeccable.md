## Design Context

### Users
SICK is used by operators, hosting providers, and technically fluent customers who need to inspect Linux server hardware and compare I/O performance from a terminal session. The primary job is to produce a trustworthy, shareable server health and benchmark report quickly, especially when evaluating disks, mounts, SMART health, and write/read performance.

### Brand Personality
Formal, reliable, and benchmark-oriented. The project can keep its playful SICK name in documentation, but the report output itself should feel like a conservative customer audit artifact rather than a decorative terminal toy.

### Aesthetic Direction
Use a restrained terminal-report style: clear section rhythm, compact tables for comparable data, explicit status wording, and minimal color used only for hierarchy or risk. Optimize for copy/paste, screenshots, and archived logs. Avoid noisy banners, excessive emoji, and repetitive per-item blocks when tabular comparison is clearer.

### Design Principles
- Lead with summary and status, then details; users should see enabled tests, tool availability, and risk at a glance.
- Prefer aligned tables for comparable resources such as mounts, disks, and I/O benchmark block sizes.
- Use generous separation between sections but dense, consistent rows within data tables.
- Never hide side effects: read/write I/O tests must be explicit, labeled, and easy to invoke from `curl | bash`.
- Preserve JSON as the stable machine-readable output while making terminal text suitable for customer-facing audit screenshots.
